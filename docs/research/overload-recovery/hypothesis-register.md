# 假设登记表 (Hypothesis Register)

> **产品**: 负荷与恢复助手 (Overload Recovery Assistant)
> **更新日期**: 2026-07-15
> **假设总数**: 6 条核心产品假设
> **评估方法**: 基于已收集的 A类学术研究、B类用户声音、C类竞品审查进行证据支持度评估

---

## 使用说明

- **现有证据支持度** 分为三级：
  - `无直接证据` — 现有证据无法直接支持或反驳该假设，需真人验证
  - `部分支持` — 有相关研究指向类似结论，但未直接验证本假设的具体机制
  - `间接支持` — 有研究或用户声音从侧面支持假设的合理性
- 每条假设标注关联证据编号（R-xxx 研究论文，VC-xxx 用户声音，CR-xxx 竞品审查）
- 验证方法指 MVP 阶段需要采用的验证手段

## H-001: 用户愿意使用一键降载

| 字段 | 内容 |
|------|------|
| **假设描述** | 用户在面对认知过载时，愿意通过一个简单的"一键降载"操作（如快速记录所有待办、清空当前视图、隐藏非紧急任务）来减轻心理负荷 |
| **假设类型** | 功能采用假设 |
| **关联证据** | VC-001（信息过载导致一事无成）、VC-003（待办列表越长越焦虑）、VC-012（待办清单越来越长）、VC-019（效率工具增加负担）、CR-Todoist（任务堆积无降载机制） |
| **现有证据支持度** | **无直接证据** |
| **支持度说明** | 现有用户声音表明用户对"过载"有明确感知和痛苦，且现有工具缺乏降载机制（竞品审查证实）。但"用户是否愿意使用一键降载"这一具体行为假设，尚无直接研究或用户声音验证。用户可能更倾向于"完成"而非"降载"——降载行为本身可能引发"逃避"的负面自我评价。 |
| **需填补的缺口** | GAP-011: 一键降载功能的用户接受度验证（P0优先级） |
| **验证方法** | MVP 原型测试：提供一键降载按钮，观察 10-15 名参与者的使用频率、使用后情绪变化（简短访谈） |
| **优先级** | P0（核心功能假设，直接决定产品价值主张） |
| **验证时机** | MVP 原型阶段（Day 0 观察首次使用） |

## H-002: 深夜隐藏任务压力会降低焦虑

| 字段 | 内容 |
|------|------|
| **假设描述** | 在深夜时段（22:00-06:00）自动隐藏未完成任务列表，仅显示恢复/睡眠相关内容，会降低用户的睡前焦虑，减少报复性睡前拖延 |
| **假设类型** | 机制假设 |
| **关联证据** | R-001（睡前拖延与压力关联 r=0.38，元分析 N=35097）、R-002（睡前拖延横断面研究 N=741）、VC-004（睡前拖延脑子停不下来）、VC-005（睡眠追踪加重焦虑）、VC-018（报复性睡前拖延）、VC-022（睡眠数据引发焦虑） |
| **现有证据支持度** | **部分支持** |
| **支持度说明** | R-001 元分析（N=35097）证实睡前拖延与日间压力/疲劳显著正相关（r=0.38），表明任务压力确实是睡前拖延的驱动因素之一。R-002 进一步确认了这一关联。用户声音 VC-004、VC-018 描述了"脑子停不下来"的具体体验，与任务压力未释放一致。但"隐藏任务"是否能有效降低焦虑——而非只是推迟焦虑——尚无直接证据。存在风险：用户可能在隐藏后通过其他渠道（如回忆）继续焦虑，或对"被系统隐藏"产生不信任感。VC-005 提示，用户对系统干预睡眠行为可能产生反向焦虑。 |
| **需填补的缺口** | GAP-012: 深夜隐藏任务对焦虑水平的实际影响测量（P0优先级） |
| **验证方法** | MVP 测试：A/B 对比组（隐藏 vs 不隐藏），使用简短焦虑自评（1-5分）在睡前和晨起记录，7天周期 |
| **优先级** | P0（核心差异化功能，直接关联"恢复"价值主张） |
| **验证时机** | MVP 测试阶段（Day 1-7） |

## H-003: Waiting For 会形成持续负荷

| 字段 | 内容 |
|------|------|
| **假设描述** | 等待他人回复或行动的事项（Waiting For）会在心理上形成持续的认知负荷，即使这些事项不在当前行动列表中，也会消耗用户的注意力资源 |
| **假设类型** | 机制假设 |
| **关联证据** | R-003（工作边界与恢复前瞻性纵向研究 N=337/169）、VC-001（信息过载导致一事无成）、VC-012（待办清单越来越长）、CR-Todoist（无 Waiting For 隔离机制）、CR-Notion（所有信息混在一起增加认知负荷） |
| **现有证据支持度** | **间接支持** |
| **支持度说明** | R-003 前瞻性纵向研究（N=337基线/169随访）证实，心理脱离（psychological detachment）困难与工作-家庭冲突、恢复不足显著相关。Waiting For 事项本质上是阻碍心理脱离的因素——它们是"未完成的心理循环"（Zeigarnik 效应）。用户声音 VC-001、VC-012 描述的"脑子同时运行太多未完成的事"与 Waiting For 负荷的机制一致。竞品审查发现，主流待办应用缺乏对 Waiting For 的专门管理，导致其混入主任务列表加剧负荷。但"Waiting For 独立于主任务列表形成持续负荷"这一具体机制，尚需直接验证。 |
| **需填补的缺口** | GAP-013: Waiting For 事项对认知负荷的独立贡献测量（P1优先级） |
| **验证方法** | MVP 测试：对比有/无 Waiting For 分区管理的用户，使用认知负荷量表（NASA-TLX 简版）和主观恢复感评分 |
| **优先级** | P1（重要功能假设，但非 MVP 首要验证目标） |
| **验证时机** | MVP 测试阶段（Day 1-7），可在 H-002 验证后补充 |

## H-004: 十秒签到的留存高于长篇日记

| 字段 | 内容 |
|------|------|
| **假设描述** | 提供10秒内可完成的极简签到（如选择一个状态标签 + 可选一句话备注）的用户留存率，显著高于需要撰写长篇日记的用户留存率 |
| **假设类型** | 产品设计假设 |
| **关联证据** | R-007（健康应用弃用全国调查 N=1604）、R-008（健康应用弃用纵向定性研究 N=17）、VC-003（待办app用三天就放弃）、VC-019（效率工具维护成本高）、VC-021（番茄钟应用太复杂）、CR-Daylio（极简签到模式的成功案例）、CR-Moodnotes（复杂交互导致弃用） |
| **现有证据支持度** | **间接支持** |
| **支持度说明** | R-007 全国调查（N=1604）明确指出"数据输入负担过大"是健康应用被弃用的首要原因（占比最高）。R-008 纵向定性研究（N=17）进一步揭示，用户嫌弃需要持续手动输入的应用，转向更轻量的替代品。用户声音 VC-003、VC-019、VC-021 一致表明功能复杂度和使用成本是弃用的直接原因。竞品审查中 Daylio 的极简情绪签到模式获得了较高的用户留存评价，而 Moodnotes 因交互复杂被批评。这些证据从侧面支持"签到越简留存越高"，但"10秒"这一具体阈值尚无直接验证，且不同用户群体对"极简"的接受度可能不同。 |
| **需填补的缺口** | GAP-014: 签到时长与留存率的定量关系验证（P1优先级） |
| **验证方法** | MVP 测试：A/B 对比组（10秒极简签到 vs 30秒简短日记 vs 2分钟完整日记），追踪 7 日留存率 |
| **优先级** | P1（影响核心交互设计，但 MVP 可先用极简方案验证） |
| **验证时机** | MVP 测试阶段（Day 1-7 留存追踪） |

## H-005: 用户愿意记录大概睡眠时间

| 字段 | 内容 |
|------|------|
| **假设描述** | 用户愿意每天手动记录"大概的"睡眠时间（如"昨晚11点半到今早7点"），而不需要精确的睡眠追踪设备或自动监测 |
| **假设类型** | 功能采用假设 |
| **关联证据** | R-005（睡眠应用系统评估73个应用）、R-006（睡眠应用综述）、R-007（健康应用弃用调查 N=1604）、VC-005（睡眠追踪加重焦虑）、VC-022（睡眠数据太多反而焦虑）、CR-SleepCycle（自动监测不准被批评）、CR-Pillow（精确数据引发焦虑） |
| **现有证据支持度** | **间接支持** |
| **支持度说明** | R-007（N=1604）表明用户弃用健康应用的首要原因是"数据输入负担过大"，但同时 R-005 评估发现，手动输入在睡眠应用中是被接受的模式——关键在于输入的复杂度。VC-005 和 VC-022 揭示了精确自动监测的负面效果：用户因"睡眠质量差"的评分而焦虑，反而影响睡眠。这间接支持"大概记录优于精确追踪"的设计方向。竞品审查中 Sleep Cycle 和 Pillow 的自动监测因不准和引发焦虑被批评。但"用户是否愿意每天手动记录"——即使是大概时间——仍需验证，因为手动记录本身也是一种使用负担。 |
| **需填补的缺口** | GAP-015: 手动大概睡眠时间记录的持续使用意愿验证（P1优先级） |
| **验证方法** | MVP 测试：提供极简睡眠时间选择器（起床时一键选择"早睡/正常/晚睡/熬夜"），追踪 7 日使用率 |
| **优先级** | P1（核心数据输入方式，影响恢复评估功能） |
| **验证时机** | MVP 测试阶段（Day 1-7 使用率追踪） |

## H-006: 用户接受系统建议延期任务

| 字段 | 内容 |
|------|------|
| **假设描述** | 当系统基于用户当前负荷状态建议延期非紧急任务时，用户会接受建议而非感到被干预或被评判 |
| **假设类型** | 机制假设 |
| **关联证据** | R-004（数字健康产品采用障碍 PPI定性研究 N=118）、VC-003（待办列表越长越焦虑）、VC-012（待办清单越来越长）、CR-Todoist（无智能延期建议）、CR-TickTick（手动延期操作繁琐） |
| **现有证据支持度** | **无直接证据** |
| **支持度说明** | R-004（N=118）的 PPI 定性研究揭示了数字健康产品采用的核心障碍：用户反感被"管理"或"评判"，期望工具是辅助而非指令。这一发现提示"系统建议延期"可能被部分用户感知为"系统在告诉我做不完"的负面评判。用户声音 VC-003、VC-012 表明用户对任务堆积有焦虑，但也可能将"延期建议"视为对自己能力的否定。竞品审查发现，现有工具要么不提供延期建议（Todoist），要么需要用户手动操作（TickTick），缺乏系统主动建议的先例。因此，"用户接受系统建议"这一假设风险较高——需要精心设计建议的措辞和呈现方式，避免触发用户的抗拒反应。 |
| **需填补的缺口** | GAP-016: 系统建议延期的用户接受度和措辞敏感性测试（P0优先级） |
| **验证方法** | MVP 原型测试：提供不同措辞的延期建议（"建议延期"vs"你看起来很忙，要不要把这件事挪到明天？"），观察接受率和情绪反应 |
| **优先级** | P0（高风险假设，措辞不当可能导致用户流失） |
| **验证时机** | MVP 原型阶段（Day 0 观察首次反应） |

## 验证路线图

### 阶段一: MVP 原型阶段（Day 0）

**验证目标**: H-001（一键降载接受度）、H-006（系统建议延期接受度）

**方法**: 
- 10-15 名参与者在观察下首次使用产品
- 记录对一键降载按钮的第一反应（是否点击、点击后表情/语言反馈）
- 记录对系统延期建议的第一反应（接受/拒绝/犹豫，及语言反馈）
- 观察后简短访谈（5分钟），询问感受和建议

**通过标准**:
- H-001: ≥60% 参与者在首次过载场景下主动使用一键降载
- H-006: ≥50% 参与者接受至少一次系统延期建议，且无明确负面情绪表达

### 阶段二: MVP 测试阶段（Day 1-7）

**验证目标**: H-002（深夜隐藏降焦虑）、H-003（Waiting For 负荷）、H-004（签到留存）、H-005（睡眠记录意愿）

**方法**:
- 自然使用 7 天，每日签到 + 睡眠记录
- H-002: A/B 对比（隐藏 vs 不隐藏），睡前/晨起焦虑自评
- H-003: 对比有/无 Waiting For 分区的认知负荷评分
- H-004: A/B 对比不同签到方式，追踪 7 日留存
- H-005: 追踪每日睡眠记录完成率
- Day 7 深度访谈（30分钟）

**通过标准**:
- H-002: 隐藏组焦虑自评降幅 ≥ 不隐藏组，且无"隐藏后更焦虑"的报告
- H-003: 有 Waiting For 分区组认知负荷评分低于无分区组
- H-004: 极简签到组 7 日留存率 ≥ 长篇日记组
- H-005: 7 日睡眠记录完成率 ≥ 70%

### 阶段三: 上线后持续验证

**验证目标**: 所有假设的长期稳定性

**方法**:
- 产品内置匿名使用数据采集（需用户授权）
- 月度留存与功能使用率追踪
- 季度用户满意度调查

## 风险标注

| 假设 | 主要风险 | 风险等级 | 缓解措施 |
|------|----------|----------|----------|
| H-001 | 用户将"降载"视为"逃避"产生负面自我评价 | 中 | 措辞设计为"暂存"而非"放弃"，强调恢复而非逃避 |
| H-002 | 隐藏任务后用户通过回忆继续焦虑，或对系统隐藏产生不信任 | 高 | 提供明确的"明天可以看到"承诺，允许手动查看 |
| H-003 | Waiting For 分区反而让用户更关注等待事项 | 低 | 分区设计为"收起"状态，默认不展开 |
| H-004 | 极简签到导致数据不足以提供有价值的恢复建议 | 中 | 允许可选的详细模式，极简为默认 |
| H-005 | 手动记录遗忘率高，数据断续 | 中 | 晨起推送提醒，一键选择降低成本 |
| H-006 | 系统建议被感知为"评判"或"指令" | 高 | 使用温和措辞，提供"不用了"选项，从不强制 |

## 证据关联说明

本登记表的证据支持度评估基于以下已收集证据：

- **A类研究**: 10 篇真实学术论文（详见 `public-research-register.md`）
- **B类用户声音**: 22 条真实公开用户声音（详见 `voice-of-customer-register.md`）
- **C类竞品审查**: 18 个产品的真实审查（详见 `competitor-review-register.md`）
- **证据矩阵**: 详见 `evidence-matrix.md`
- **证据缺口**: 详见 `evidence-gaps.md`

证据支持度的升级路径：
1. `无直接证据` → 通过 MVP 验证收集 D 类证据 → 升级为 `部分支持` 或 `反驳`
2. `部分支持` → 通过更多相关研究或用户验证 → 升级为 `间接支持` 或 `直接支持`
3. `间接支持` → 通过直接验证 → 升级为 `直接支持` 或 `反驳`
