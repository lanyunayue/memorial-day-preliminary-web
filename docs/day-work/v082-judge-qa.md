# 时刻 v0.8.2 评委 Q&A

1. 为什么不直接用 Google Calendar？
   Google Calendar 适合正式日程。时刻处理的是更早一步：把一句话先整理成结构化记录，再决定是否导入日历。

2. 为什么不直接用 Days Matter？
   Days Matter 更偏纪念日和倒计时。时刻同时覆盖提醒、纪念、习惯、备忘，并强调一句话输入和日历/备份衔接。

3. 提醒关闭网页后还能响吗？
   不保证。网页提醒依赖浏览器环境。重要日程建议导出 `.ics` 到系统日历。

4. 清缓存数据会不会丢？
   有可能。时刻的数据保存在当前浏览器，所以 v0.8.2 提供 JSON 备份和导入预览。

5. 为什么不做云同步？
   云同步需要账号、后端、隐私和稳定性设计。复赛候选先把本地数据安全和标准文件导出做扎实。

6. 为什么不做登录？
   登录会增加复杂度和隐私责任。当前版本强调免登录、低门槛、本地使用。

7. `.ics` 是什么？
   `.ics` 是标准日历文件格式，可手动导入 Google Calendar、Apple Calendar 或 Outlook。

8. 手动导入麻烦怎么办？
   这是当前 Web 版的边界。好处是不需要授权账号，也不接第三方 API，风险更低。

9. NLP 识别错怎么办？
   用户可以编辑记录。复赛后应继续扩展 NLP 测试集和编辑纠错体验。

10. 数据安全怎么保证？
    当前是本地浏览器存储，不等同云备份。v0.8.2 通过 JSON 备份、导入预览、id 修复和失败回滚降低风险。

11. 时刻精灵有什么实际价值？
    它降低首次使用理解成本，引导用户写一句话或生成示例，不承担核心功能。

12. 这是产品还是 demo？
    它已经有完整的记录、搜索、筛选、日历、导出、备份和测试链路，但仍是 Web 复赛候选，不是商业化完整版。

13. 技术创新在哪里？
    创新点不是炫技，而是把一句话输入、时间结构化、本地数据保护和标准日历衔接做成一个闭环。

14. 工程质量怎么证明？
    有回归套件、ICS 深度测试、备份安全测试、导入预览测试、PWA 诚实说明测试和 Edge CDP 运行态验收。

15. 测试做了什么？
    覆盖 PWA、HTML、A11y、响应式、i18n、体验示例、时刻精灵、ICS、备份、导入预览、数据安全中心和提醒说明。

16. 如何继续迭代到 HarmonyOS？
    可先保持同样的数据模型和导出格式，再做原生提醒、桌面组件和系统日历联动。

17. 和手机自带日历怎么协同？
    时刻负责前置整理，自带日历负责系统级提醒。`.ics` 是两者之间的低风险桥梁。

18. 用户为什么会留下？
    因为入口比日历轻，记录比备忘录有时间感，又能导出到正式日历和备份。

19. 如何处理隐私？
    当前数据默认保存在本地浏览器，不上传服务器。不做云同步也减少了隐私面。

20. 复赛后最优先做什么？
    优先做更完整的 NLP 回归、跨设备数据方案、原生提醒能力评估和更清晰的导入导出引导。

21. `.ics` 会自动同步吗？
    不会。它是手动导入文件。当前版本不写自动同步，也不接 Google Calendar API。

22. JSON 备份和 `.ics` 有什么区别？
    JSON 用于恢复时刻数据；`.ics` 用于导入第三方日历。两者不是同一种备份。

23. 为什么不用大模型 API？
    当前 NLP 规则足够覆盖演示核心场景。接 API 会引入成本、网络、隐私和稳定性问题。

24. 如果用户导入坏文件怎么办？
    坏 JSON 不会写入；导入前会展示预览，确认后才写入。

25. 当前最大短板是什么？
    Web 版本地存储和提醒能力天然有限。v0.8.2 已经把这个边界说明清楚，并提供可行的备份和日历导出路径。
